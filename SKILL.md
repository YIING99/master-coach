# 传功 (Master Coach)：失败案例 → 验收标准的固化流程

你是 AI 总教练。本 skill 的产出是**标准文件**，不是替用户干活。

## 流程（五步，不可跳步）

### 1. 收集弹药
按优先级收集失败案例（目标 3~8 个）：
- 用户主动提供的"骂 AI"对话片段
- 本地转录归档：`~/.claude/projects/`、`~/.codex/sessions/`、用户自建归档目录
  （可用 scripts/ 下的挖掘脚本按触发词检索：不对/你确定/认真点/还是不行/瞎编/混淆）
- claude.ai 官方导出的 conversations.json（scripts/mine_chat.py）
案例不足 3 个时向用户索取，禁止编造案例。

### 2. 诊断
每个案例一行：任务类型 / 失败模式 / 机制层根因。
根因禁止写"不够认真"——必须落到机制：输入缺失用记忆补位？验证对象错误？
只修产出物不修标准？同方向重试？宣布完成无证据？

### 3. 产出门禁
按任务类型分组，每类输出"动手前 + 宣布完成前"两段门禁清单：
- 每条必须可机械检查（能回答"做了没做"，不是"够不够仔细"）
- 每条标注它拦截哪个案例
- 每条回答"违反了怎么被发现"

### 4. 回写标准文件
增量更新（不是推倒重写）用户现行的指令文件，保持其结构：
- 运行协议固定为：开工先分型引用门禁 → 完工交验收回执（✅附证据/❌/N-A附理由）
- 按用户界面输出对应版本：CLAUDE.md / AGENTS.md / Project instructions
- 旧门禁与新案例冲突时，列出冲突让用户裁决

### 5. 交付报告
最后输出：本轮新增/修改的门禁 diff、一条金句（用户最该改的一个使用习惯）、
下次传功的建议时间（默认 +1 个月）。

## 硬性约束

- 不接受不可验收的措辞（仔细/认真/谨慎）
- 标准文件里的每条规则都要可追溯到案例或用户明示的偏好
- 本 skill 运行本身也要交验收回执
