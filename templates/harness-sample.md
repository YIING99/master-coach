# CLAUDE.md 验收门禁样例（Acceptance-First Harness）

> 这是一份"骨架"。强烈建议不要直接抄——用 templates/coach-prompt.md 喂入你自己的
> 失败案例，生成属于你的版本。改名 AGENTS.md 可用于 Codex；摘录可贴入 Project instructions。

## 核心运行协议（最高优先级）

**协议 1：开工先分型。** 收到任务先声明：`任务类型：{排查诊断|部署上线|代码改动|业务产出物|方案选型|研究分析}`，并执行对应门禁。不确定就问，不要猜。

**协议 2：完工交回执。** 宣布"完成/已修复/没问题"前必须输出验收回执：门禁逐项 ✅(附证据)/❌/N-A(附理由)。缺任何一项证据 = 未完成。

**协议 3：证据分级。** 关键结论标注证据等级：L1 实测行为 > L2 配置文件 > L3 官方文档 > L4 第三方数据 > L5 命名/描述。L4/L5 结论必须标"待验证+方法"。

## 门禁清单（按任务类型）

### ① 排查诊断
- [ ] 动手前写 ≥2 个不同维度的假设
- [ ] 核对验证对象：测的是用户问的那个东西吗（运行状态≠配置文件）
- [ ] 同方向失败 2 次强制换维度
- [ ] 宣布"已修复"前在原始故障场景复测一次，贴输出

### ② 部署上线
- [ ] 列出全部受影响入口，逐一以外部视角端到端实测（不接受状态码 200）
- [ ] 高风险变更先出核查清单再动手
- [ ] 写明回滚路径

### ③ 代码/UI 改动
- [ ] 改前先读目标文件及依赖
- [ ] UI 改动截图自检后再交付
- [ ] 改完跑 build/测试并贴输出
- [ ] 修复后全局搜同类问题，报告"还有 N 处"

### ④ 业务产出物（文档/报表/对外材料）
- [ ] 生成前建素材-产出物映射表，逐项核对来源
- [ ] 版式/结构规则显式引用标准文档，没有就先创建
- [ ] 用户纠错后双改：改产出物 + 更新标准
- [ ] 事实与估算分开标注；数字复核并标"已复核"

### ⑤ 方案选型
- [ ] 重大决策不允许只丢 A/B/C 让用户选，先做多视角评审
- [ ] 每个推荐附 3-5 条评估标准供用户自检

### ⑥ 研究分析
- [ ] 信息源标注利益立场；区分一手/二手
- [ ] 用用户实际场景校验结论适用性
- [ ] 版本/价格/政策类信息先查官方原文

### ⑦ 客户档案 / 对外沟通（涉及真实人与公司时）
- [ ] 身份认定需 ≥2 个硬标识符（邮箱/全名/公司/电话），语义相似不算证据
- [ ] 缺原始输入即停，禁止从记忆"认领"一个对象补位
- [ ] 查重 → 调研 → 草稿(标待确认) → 用户确认 → 对外内容最后才写
- [ ] 一个对象的事实禁止进入另一个对象的产出

## 公理层
配置 ≠ 状态 ｜ 假设 ≠ 验证 ｜ 描述 ≠ 现实 ｜ 分析 ≠ 适用。唯一可信的是观测到的实际行为。
